3. SEQUENCE DIAGRAM - AUDIO SUBMISSION & ASSESSMENT (Page 5)
================================================================================

PARTICIPANTS (left to right):
-----------------------------
1. Student
2. Web App (React)
3. API Server (Node.js)
4. Database (PostgreSQL)
5. Cloud Storage (R2/S3)

MESSAGE FLOW:
-------------

[Student Submission Phase]
1. Student -> Web App: Record/Upload Audio
2. Web App -> API Server: POST /api/submissions {task_id, audio_file}
3. API Server -> Cloud Storage: Upload audio to cloud storage (MP3 conversion)
4. Cloud Storage -> API Server: Return audio_url
5. API Server -> Database: INSERT MemorizationSubmission (task_id, audio_url, status=pending)
6. Database -> API Server: Confirm submission stored
7. API Server -> Web App: Return 201 Created {submission_id, status}
8. Web App -> Student: Display "Pending Review"

[Teacher Assessment Phase] (async, in reference frame)
9. API Server -> Database: Teacher queries pending submissions
10. Database -> API Server: Return pending submissions list with audio URLs
11. API Server -> Web App: GET /api/submissions/:id (audio stream)
12. Web App -> Student: Play audio with timestamped comments
13. Student -> Web App: Submit assessment scores (mem, tajweed, fluency)
14. Web App -> API Server: POST /api/assessments {scores, feedback, timestamps}
15. API Server -> Database: INSERT Assessment + UPDATE Submission status
16. Database -> API Server: Confirm assessment saved

4. ACTIVITY DIAGRAM - STUDENT EXAM TAKING PROCESS (Page 6)
================================================================================

SWIMLANES (4 partitions):
-------------------------
- Student (leftmost, light green)
- Web Client (second, cream)
- API Server (third, light blue)
- Database (rightmost, light pink)

FLOW:
-----
[Start] -> Student: Login & Authenticate
  -> Web Client: Validate JWT Token
    -> API Server: Verify RBAC Permissions
      -> Database: Verify Token & Role

-> Student: Navigate to Exam Dashboard
  -> Web Client: Fetch Exam List
    -> API Server: Query Active Exams
      -> Database: SELECT Exams & Questions

-> [Decision: Exam Available?]
  [No] -> Loop back to Navigate to Exam Dashboard
  [Yes] -> Student: Select Available Exam
    -> Web Client: Load Exam Metadata
      -> API Server: Validate Time Window
        -> Database: Check Enrollment & Attempts

-> Student: Read Instructions & Start Exam
  -> Web Client: Render Timer & Question UI
    -> API Server: Serve Questions (Randomized)
      -> Database: Log Start Timestamp

-> [Decision: Time Remaining?]
  [No] -> Auto-Submit on Timeout
  [Yes] -> Student: Answer Questions (Navigate)
    -> Web Client: Cache Answers (LocalStorage)
      -> API Server: Auto-Submit on Timeout
        -> Database: Store Answers Temporarily

-> Student: Review Answers & Submit
  -> Web Client: POST Final Submission
    -> API Server: Auto-Grade MCQ/TF
      -> Database: INSERT Submission & Answers

-> [Decision: Short Answer?]
  [Yes] -> Flag for Manual Grading (Teacher reviews later)
  [No] -> API Server: Calculate Statistics
    -> Database: UPDATE Results & Analytics

-> Student: View Results & Feedback
  -> Web Client: Display Score & Analytics
    -> [End]

6. STATE DIAGRAM - MEMORIZATION SUBMISSION LIFECYCLE (Page 8)
================================================================================

STATES:
-------
[Initial State] -> ASSIGNED
  (Task created by teacher)

ASSIGNED -> IN_PROGRESS
  Event: student starts recording

IN_PROGRESS (self-loop)
  Event: recording... (ongoing)

IN_PROGRESS -> SUBMITTED
  Event: upload complete [guard: audio < 20MB]
  [Guard condition: file_size <= 20MB]

IN_PROGRESS -> ASSIGNED
  Event: cancel / timeout

SUBMITTED -> PENDING_REVIEW
  Event: queued for teacher review

PENDING_REVIEW -> UNDER_REVIEW
  Event: teacher opens audio player

UNDER_REVIEW (self-loop)
  Event: listening / scoring (ongoing)

UNDER_REVIEW -> EVALUATED
  Event: assessment submitted [scores >= threshold]
  [Guard: overall_score >= 60%]

UNDER_REVIEW -> REQUIRES_REVISION
  Event: assessment submitted [scores < threshold]
  [Guard: overall_score < 60%]

REQUIRES_REVISION -> IN_PROGRESS
  Event: student resubmits revised audio

EVALUATED -> ARCHIVED
  Event: 30 days after assessment

ARCHIVED -> [Final State]
  (Retained for 1 year)

9. COMMUNICATION DIAGRAM - LIVE CLASS SESSION (WEBRTC) (Page 11)
================================================================================

FRAME: "Live Session Lifecycle"

OBJECTS (Participants):
-------------------------
- Teacher Client (top left)
- Signaling Server (top center)
- Daily.co / Twilio (top center-right)
- Student Client A (top right)
- Student Client B (bottom left)
- Recording Service (bottom center)
- Chat Server (bottom center-right)
- Attendance Tracker (bottom right)

NUMBERED MESSAGE SEQUENCE:
--------------------------
1. Teacher Client -> Signaling Server: createRoom() {session_id, max_participants}
2. Signaling Server -> Daily.co/Twilio: createMeeting() {teacher_token, config}
3. Daily.co/Twilio -> Signaling Server: meeting_url + room_token
4. Signaling Server -> Teacher Client: join_url + host_controls
5. Signaling Server -> Student Client A: notify() {join_url, 15min_before}
6. Student Client A -> Daily.co/Twilio: join() {student_token, session_id}
7. Daily.co/Twilio -> Student Client A: peer_connection established (WebRTC)
8. Teacher Client -> Student Client A: audio/video stream (RTP)
9. Student Client A -> Teacher Client: audio/video stream (RTP)
10. Teacher Client -> Chat Server: sendMessage() {text, timestamp}
11. Chat Server -> Student Client A: broadcast() {message, sender}
12. Chat Server -> Student Client B: broadcast() {message, sender}
13. Daily.co/Twilio -> Recording Service: startRecording() {session_id}
14. Daily.co/Twilio -> Attendance Tracker: trackJoin() {user_id, timestamp}
15. Student Client B -> Daily.co/Twilio: join() {student_token}
16. Teacher Client -> Signaling Server: endSession() {session_id}
17. Signaling Server -> Daily.co/Twilio: closeRoom() {session_id}
18. Daily.co/Twilio -> Recording Service: stopRecording() + upload_url
19. Daily.co/Twilio -> Attendance Tracker: trackLeave() {user_id, duration}
