Page 6 of 39 1. Introduction  
This Software Requirements Specification (SRS) document provides a comprehensive description of 
the Quran Learning Platform, an online educational system designed for Quran schools, teachers, 
and students. The platform facilitates Quran memorization (Hifz), Tajweed instruction, examination 
administration, progress tracking, live virtua l classes, and Islamic educational content delivery.  
This document is intended for the development team, project stakeholders, quality assurance 
engineers, and system architects. It serves as the authoritative reference for all functional and 
nonfunctional  requirements, external interfaces, and system constraints.  
1.1 Purpose  
The purpose of this SRS is to define the complete set of requirements for the Quran Learning 
Platform. It establishes the functional behavior, performance expectations, security requir ements, 
and technical constraints that shall govern the design, implementation, and validation of the system.  
This document shall be used as the basis for: software design and architecture decisions; 
development task planning and estimation; test case desi gn and acceptance criteria; and contract 
agreements between the client and development vendor.  
1.2 Scope  
The Quran Learning Platform is a web -based educational management system with planned mobile 
application support. The system shall provide the following core capabilities:  
 User authentication and role -based access control for Administrators, Teachers, and 
Students  
 Student enrollment, profile management, and progress tracking  
 Daily memorization (Hifz) task assignment and revision scheduling  
 Examin ation creation, administration, and automated grading  
 Audio -based memorization submission and teacher assessment  
 Live virtual classroom sessions for memorization and Tajweed  
 Tajweed lesson content with multimedia resources  
 Islamic educational content inclu ding supplications, Seerah, and companion stories  
 Notification system for announcements, reminders, and private messages

2. Overall Des cription  
2.1 Product Perspective  
The Quran Learning Platform is a standalone web application with a planned native mobile 
companion. It operates as a centralized educational management hub connecting three primary 
stakeholder groups: school administrators who configure and oversee the platform, teachers who 
deliver instruction and assess student performance, and students who engage with learning 
materials and submit assignments.  
The platform shall integrate with third -party services for real -time communicat ion (live classes), 
push notifications, and cloud -based media storage. It maintains its own identity provider for 
authentication but may be extended to support OAuth 2.0 providers in future releases.

Page 10 of 39 2.3 User Classes and Characteristics  
The system supports three primary user classes, each with distinct roles, privileges, and interaction 
patterns:  
2.3.1 Administrator  
The Administrator has full system access and is responsible for platform governance. Administrators 
are typically school principals, IT directors, or designated platform managers. They require 
comprehensive reporting capabilities and bulk operation tools. Key characteristics include:  
 Full CRUD (Create, Read, Update, Delete) permissions on all user accounts  
 Content management  authority across all educational modules  
 System configuration and platform setting control  
 Access to aggregate statistics and analytics dashboards  
 Announcement broadcasting to all user classes  
2.3.2 Teacher  
Teachers are the primary content creators and ev aluators. They manage student cohorts, assign 
memorization tasks, conduct examinations, and provide formative feedback. Key characteristics 
include:  
 Student management within assigned classes or groups  
 Memorization task creation and assignment  
 Examination authoring with multiple question types  
 Audio submission review and Tajweed assessment  
 Live class hosting and session management  
 Direct messaging with students  
2.3.3 Student  
Students are the primary consumers of educational content. They access assigned tas ks, submit 
memorization recordings, take examinations, and engage with Islamic educational materials. Key 
characteristics include:  
 Personal dashboard with daily task overview  
 Memorization audio recording and upload

2.4 Operating Environment  
The system shall operate within the following technical environment : 
 Server Infrastructure: Cloud -hosted (Render or Railway), Linux -based containers  
 Database: PostgreSQL 15+ with full Unicode support for Arabic script  
 Application Server: Node.js 20 LTS with Express.js framework  
 Client Browsers: Chrome 110+, Firefox 120+, Safari 17+, Edge 120+  
 Mobile: iOS 16+, Android 13+ (via planned React Native application)  
 Network: HTTPS/TLS 1.3 required for all communications  
 Media Storage: Cloud object storage for audio recordings and video content  
2.5 Design and Implementation Constraints  
The following constraints shall govern the system design and implementation:  
 The backend API must be built using Node.js with Express.js and Drizzle ORM  
 The frontend web application must use React with Tailwind CSS  
 The database must be PostgreS QL with schema managed through Drizzle migrations  
 Authentication must use JWT tokens with secure httpOnly cookie storage  
 Push notifications must use Firebase Cloud Messaging (FCM)

3. System Features  
This section provides detailed functional requirements for ea ch system feature. Each requirement is 
uniquely identified using the format [F -XXX] where F indicates a functional requirement and XXX is a 
sequential number. Priority levels are defined as: Critical (must have for MVP), High (important for 
initial release ), Medium (desirable within first release), Low (future enhancement).  
3.1 Authentication and Authorization  
The Authentication and Authorization module provides secure identity verification, session 
management, and role -based access control for all system u sers.  
3.1.1 User Registration  
F-001 [ Critical ] The system shall allow Administrator users to create Teacher and Student accounts 
with email, password, full name, and role assignment.  
F-002 [ Critical ] The system shall validate email uniqueness across all us er accounts.  
F-003 [ Critical ] Passwords shall be hashed using bcrypt with a minimum cost factor of 12 before 
storage.  
F-004 [ High ] The system shall support self -registration for Students with teacher approval workflow.  
F-005 [ High ] Registration shall include optional fields: phone number, date of birth, and profile photo.  
3.1.2 User Login  
F-006 [ Critical ] The system shall authenticate users via email and password.  
F-007 [ Critical ] Upon successful authentication, the system shall issue a JWT access token (15 -minute 
expiry) and a refresh token (7 -day expiry).  
F-008 [ Critical ] The system shall implement token refresh mechanism to maintain user sessions 
securely.  
F-009 [ High ] The system shall implement rate limiting on login attempts (maximum 5 attempts  per 
15-minute window per IP).  
F-010 [ High ] Failed login attempts shall be logged with timestamp, IP address, and user identifier for 
security auditing.

F-011 [ Critical ] The system shall enforce role -based access control (RB AC) with three roles:

3.7 Memorization Submission and Assessment  
The Memorization Submission and Assessment module enables students to record and u pload their 
Quran recitation for teacher evaluation across multiple quality dimensions.  
3.7.1 Audio Submission  
F-087 [ Critical ] Students shall be able to record audio directly in the browser or upload pre -recorded 
audio files.  
F-088 [ Critical ] Supported au dio formats shall include: MP3, M4A, and WAV.  
F-089 [ Critical ] Maximum audio file size shall be 20MB per submission.  
F-090 [ High ] The system shall automatically convert uploaded audio to a standardized format (MP3,

F-091 [High ] Students shall be able to review their recording before submission.  
F-092 [ High ] Each submission shall be linked to the corresponding memorization task.  
3.7.2 Teacher Assessment  
F-093 [ Critical ] Teachers shall be able to listen to submitted audio recordings through an integrated 
audio player.  
F-094 [ Critical ] Teachers shall evaluate submissions across three dimensions: Memorization Quality

5.1 Performance Requirements  
NF-001 [ Critical ] The system shall support a minimum of 1000 concurrent users without performance 
degradation.  
NF-002 [ Critical ] API response time for standard CRUD operations shall not exceed 500ms under 
normal load (P95).  
NF-003 [ Critical ] Page initial load time shall not exceed 3 seconds on a 10 Mbps connection.  
NF-004 [ High ] Audio file upload shall complete within 30 s econds for a 10MB file on a 5 Mbps upload 
connection.  
NF-005 [ High ] Video streaming shall start within 5 seconds and maintain a buffer of at least 5 seconds.  
NF-006 [ High ] Live class latency shall not exceed 500ms for audio and 150ms for video (teacher to 
student).  
NF-007 [ High ] Dashboard data queries shall complete within 2 seconds for date ranges up to 90 days.  
NF-008 [ Medium ] Search queries across Quranic text shall return results within 1 second.  
NF-009 [ Medium ] The system shall handle examination submi ssion spikes of up to 200 concurrent 
submissions without data loss.  
5.2 Security Requirements  
NF-010 [ Critical ] All passwords shall be hashed using bcrypt (cost factor >= 12); plaintext password 
storage is prohibited.  
NF-011 [ Critical ] JWT tokens shall be signed using RS256 with 2048 -bit keys; token secrets shall be 
rotated every 90 days.  
NF-012 [ Critical ] All API endpoints shall validate authentication tokens on every request except public 
endpoints.  
NF-013 [ Critical ] The system shall enforce HTTPS for all  communications; HTTP requests shall be 
permanently redirected to HTTPS.
